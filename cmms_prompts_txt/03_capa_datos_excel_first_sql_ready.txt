# CONTEXTO GENERAL PARA TODOS LOS PROMPTS

Proyecto: Mantenimiento [Nombre Empresa]

Objetivo:
Crear un CMMS web industrial funcional, no maqueta, no prototipo visual aislado. El sistema debe funcionar inicialmente con Excel como fuente de datos, pero debe quedar preparado para migrar a SQL Server o PostgreSQL sin rehacer la lógica de negocio.

Stack obligatorio:
Backend:
- C#
- ASP.NET Core Web API
- Clean Architecture
- Entity Framework Core preparado para SQL
- Repositorios desacoplados
- Implementación inicial con Excel
- Implementación SQL preparada
- JWT Auth
- Serilog
- FluentValidation
- OpenAPI / Swagger
- Jobs programados con Quartz.NET o Hangfire
- PDF generado desde plantillas HTML configurables
- Adaptadores para correo y SharePoint

Frontend:
- React
- TypeScript
- Vite
- React Router
- TanStack Query
- TanStack Table
- Zustand o Redux Toolkit
- Tailwind CSS
- Diseño tipo Fracttal
- Modo claro y modo oscuro
- Responsive
- Preparado para PWA
- IndexedDB para offline

Infraestructura:
- Docker Compose
- Backend
- Frontend
- Nginx
- Mailhog o simulador de correo
- Carpeta local simulada para SharePoint
- Scripts de seed/importación
- README técnico
- Despliegue Linux-ready

Reglas generales:
- No entregar solo pseudocódigo.
- Todo debe quedar integrado al repositorio.
- Cada módulo debe incluir backend, frontend cuando aplique, validaciones, pruebas básicas y documentación.
- La fuente de datos inicial será Excel.
- Debe existir una capa clara para reemplazar Excel por SQL.
- Marcar en código y documentación dónde se modifica para usar SQL.
- No hardcodear rutas, usuarios, correos ni credenciales.
- Usar variables de entorno y archivos de configuración.
- Mantener auditoría completa.
- Todo flujo crítico debe tener estados configurables.
- Todo módulo debe considerar permisos por rol y faena.
- Técnicos solo ven sus OT asignadas.
- Supervisores ven su faena.
- Bodega ve todas las bodegas.
- Costos solo visibles para roles autorizados.
- Importaciones Excel requieren validación y aprobación.
- Campos validados quedan bloqueados y solo se corrigen con auditoría.
- SharePoint y correo deben quedar con adaptadores configurables.
- El sistema debe generar PDFs desde plantillas HTML configurables.
- El sistema debe tener dashboards por rol con widgets configurables.
- Power BI debe consumir datasets/vistas de reporting, no tablas operacionales.


# PROMPT 03 — CAPA DE DATOS EXCEL-FIRST Y SQL-READY

Implementa la capa de acceso a datos en `MaintenanceCMMS.Infrastructure`.

## Interfaces en Application

Crear:

```csharp
IDataProvider
IRepository<T>
IUnitOfWork
IExcelRepository<T>
ISqlRepository<T>
IImportService
```

## Implementaciones

Crear:

```csharp
ExcelDataProvider
ExcelRepository<T>
SqlDataProvider
SqlRepository<T>
```

La implementación SQL puede quedar como placeholder funcionalmente documentado, pero debe estar claro dónde se conecta SQL Server o PostgreSQL.

## Librería Excel

Usar ClosedXML para leer y escribir Excel.

## Archivos Excel base

Crear archivos base en `/data/excel` si no existen:

```text
activos.xlsx
faenas.xlsx
ubicaciones_tecnicas.xlsx
usuarios.xlsx
roles.xlsx
bodegas.xlsx
repuestos.xlsx
stock_bodegas.xlsx
documentos.xlsx
proveedores.xlsx
solicitudes_repuestos.xlsx
ordenes_trabajo.xlsx
tareas_ot.xlsx
planes_preventivos.xlsx
checklists.xlsx
sistemas_componentes.xlsx
```

Cada archivo debe tener una hoja `Data`.

## ExcelSchemaRegistry

Crear `ExcelSchemaRegistry` con definición de columnas esperadas por archivo.

Debe incluir:

- Columnas obligatorias.
- Tipos esperados.
- Clave natural.
- Si permite actualización.
- Si requiere aprobación.

## Funcionalidad

El sistema debe:

- Validar columnas obligatorias.
- Crear archivos vacíos con headers si no existen.
- Leer datos.
- Guardar datos.
- Registrar errores de validación.
- Evitar duplicados por clave natural.
- Permitir cambiar proveedor de datos con configuración.

## Configuración

En appsettings:

```json
{
  "DataProvider": {
    "Provider": "Excel",
    "ExcelPath": "data/excel",
    "SqlServerConnectionString": "",
    "PostgreSqlConnectionString": ""
  }
}
```

## Endpoints de prueba

Crear:

- GET `/api/system/data-provider`
- GET `/api/system/excel-health`
- GET `/api/system/excel-schemas`

## Documentación

Crear:

```text
docs/excel-to-sql-migration.md
```

Debe explicar:

- Dónde está la implementación Excel.
- Qué interfaces se mantienen.
- Qué clases reemplazar para SQL.
- Cómo activar SQL con `DataProvider: SqlServer`.
- Cómo activar PostgreSQL.
- Qué módulos no se deben modificar al migrar.
- Qué riesgos tiene operar productivamente con Excel.

Al finalizar:
- Ejecutar tests.
- Confirmar que backend levanta con proveedor Excel.
